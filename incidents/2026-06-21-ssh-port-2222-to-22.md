# SSH Port Stuck on 2222 — Moved Back to 22

**Date:** 2026-06-21

## What happened
SSH had been changed from port 22 to 2222 at some point (likely for security-through-obscurity reasons). We wanted to move it back to 22 because:
- Port 22 was already open in UFW
- Standard port works better with tools like Claude Code web
- Hiding SSH on a weird port doesn't meaningfully stop attackers if you already have key-based auth

## Why it was tricky
Editing `/etc/ssh/sshd_config` and restarting SSH didn't work — the port stayed on 2222. That's because modern Ubuntu uses **systemd socket activation** for SSH.

With socket activation, *systemd* opens the port and hands incoming connections to sshd — sshd itself doesn't open the port. So `sshd_config` controls authentication and behavior, but the port is controlled by the systemd socket unit, not the config file.

There was a socket override file that was setting the port to 2222:
```
/etc/systemd/system/ssh.socket.d/override.conf
```
Content:
```ini
[Socket]
ListenStream=
ListenStream=0.0.0.0:2222
ListenStream=[::]:2222
```

The base socket unit (`/usr/lib/systemd/system/ssh.socket`) already had port 22 — we just needed to remove the override.

## Fix
```bash
# Confirm what sshd_config thinks the port is (should say 22 after editing)
sshd -T | grep port

# Remove the systemd socket override
rm /etc/systemd/system/ssh.socket.d/override.conf

# Reload and restart the socket (not just the service)
systemctl daemon-reload
systemctl restart ssh.socket

# Verify
ss -tlnp | grep sshd
```

## What I learned
- Editing `sshd_config` is not enough to change the SSH port on Ubuntu with systemd socket activation
- `sshd -T` shows the *effective* config as sshd reads it — useful for confirming changes took effect at the config level
- `ss -tlnp` shows what port is actually open — the ground truth
- Always check `/etc/systemd/system/*.socket.d/` for overrides if a service isn't behaving as expected

## Follow-up
- UFW had two duplicate rules for port 22 — still there, harmless but worth cleaning up eventually
- Security note: `PermitRootLogin yes` is set in sshd_config — acceptable for now since key-based auth only, but worth revisiting
